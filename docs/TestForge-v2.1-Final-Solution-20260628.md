## 一、核心发现：市场真正缺什么

经过 4 个维度（开源工具 × 商业平台 × 学术前沿 × 开发者痛点）的系统性调研，发现：

```
❌ 市场不缺: 测试框架 (pytest/Jest/JUnit/Playwright 已充分竞争)
❌ 市场不缺: 单一功能工具 (API测试/代码生成/报告 各有成熟方案)

✅ 市场真缺:
  1. 测试资产管理平台 (选择什么测试跑、隔离flaky、量化健康度) ← 最大空白
  2. 自愈测试 (不只是UI选择器，而是跨层级的自动修复) ← 商业刚起步
  3. 测试环境+数据即服务 (按需创建、生产级脱敏数据注入) ← 零产品
  4. 跨层级统一编排 (单元+集成+E2E+性能+安全 一个平面管理) ← 碎片化严重
```

## 二、TestForge v2.1 重新定位

```
v2.0: 测试生成 + 执行平台

           生成测试    跑对测试    测试不腐    一键环境    一目了然
```

## 四、v2.1 三大杀手级功能详解

### 🎯 杀手功能 #1: 测试影响分析 (TIA)

```
                    ┌──────────┐
                    │ Git Diff │
                    └────┬─────┘
                         ▼
              ┌──────────────────┐
              │ 依赖图分析        │
              │ 变更函数→调用链→  │
              │ 受影响的测试      │
              └────────┬─────────┘
                       ▼
         ┌─────────────────────────┐
         │  智能测试选择            │
         │                         │
         │  全量测试: 500个 (45min) │
         │  TIA选择:  37个  (3min) │ ← 15x加速
         │  覆盖率影响: 92% 相同    │
         └─────────────────────────┘
```

**为什么这是杀手功能**: Google/Meta 内部有但未开源。开发者最大的痛点是"改一行代码等半小时CI"。TIA 直接让 CI 时间从 45 分钟降到 3 分钟。

### 🎯 杀手功能 #2: Flaky 检测 + 自动隔离

```
        测试运行 N 次 (默认5次)
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
  全通过   全失败   有时通过有时失败
    │         │         │
    │         │         ▼
    │         │    ┌──────────┐
    │         │    │ = Flaky! │
    │         │    └────┬─────┘
    │         │         ▼
    │         │    ┌──────────────┐
    │         │    │ 自动隔离     │
    │         │    │ 移到quarantine│
    │         │    │ 通知责任人   │
    │         │    │ AI分析根因   │
    │         │    └──────────────┘
    ▼         ▼
  ✅ 正常   ❌ 真Bug
```

**为什么这是杀手功能**: Flaky tests 是开发者公认的 #1 痛点，Atlassian 为此自建了 Flakinator 平台。没有任何开源产品解决这个问题。

### 🎯 杀手功能 #3: 自愈测试 (跨层级)

```
UI 变化: 按钮 class 从 .btn-primary 变成 .btn-main
         → AI 自动更新选择器为 [data-testid="submit"]
         
API 变化: /api/v1/users → /api/v2/users, 响应新增字段
         → AI 自动更新 URL + Schema 断言
         
Schema 变化: 数据库字段 user_name → username
           → AI 自动更新测试数据 + 断言
```

**为什么这是杀手功能**: 商业平台(SmartBear/Tricentis)的 self-healing 仅限 UI。TestForge 跨 UI/API/Schema 三层自愈，业界首创。

## 五、技术栈（v2.1 最终版）

| 层 | 技术 | 理由 |
|----|------|------|
| **前端** | React 18 + TypeScript + TailwindCSS | 组件化、类型安全、快速UI |
| **后端** | FastAPI (Python 3.11+) + WebSocket | 异步高性能、实时推送、Python生态 |
| **数据库** | SQLite(本地) / PostgreSQL(服务端) | 渐进式：单文件起步→数据库扩展 |
| **任务队列** | Celery + Redis | 异步执行测试套件、分布式 |
| **LLM接口** | LiteLLM (Qwen/DeepSeek/OpenAI fallback) | 100+模型统一接口，已验证 |
| **AST解析** | tree-sitter (Python/JS/Java/Go/C++/TS) | 60+语言支持，C级性能 |
| **测试执行** | Docker + testcontainers | 沙箱隔离、即用即弃 |
| **浏览器自动化** | Playwright (Chromium/Firefox/WebKit) | 录制+回放+执行+截图 |
| **API录制** | Keploy eBPF (可选) | 零侵入API流量录制 |
| **变异测试** | PITest(Java) / Stryker(JS) / Infection(PHP) | 各语言最佳变异工具 |
| **属性测试** | Hypothesis(Python) / fast-check(JS) | 自动生成+最小反例 |
| **可观测** | OpenTelemetry + Prometheus + Grafana | 分布式追踪+指标 |
| **CI/CD** | GitHub Actions / GitLab CI / Jenkins 插件 | 全CI覆盖 |
| **部署** | Docker Compose (单机) / Helm (K8s) | 简单→企业渐进 |

## 六、核心数据结构

### 测试用例（支持全类型）

```yaml
id: "tc_order_flow_001"
name: "下单流程-库存不足"
type: functional                    # functional | boundary | api | e2e | unit | performance
tags: [order, inventory, P0]
status: active                     # active | quarantine | deprecated
flaky_score: 0.02                  # Flaky概率 (贝叶斯估计)
health_score: 92                   # 健康度评分
last_healed: null                  # 上次自愈时间
created_by: "ai"                   # ai | manual | recorded | imported

variables:
  base_url: "https://api.shop.com"
  
steps:
  - id: step1
    type: http_request
    selector:                     # 自愈用选择器（比硬编码URL更健壮）
      method: POST
      path_pattern: "/api/orders"
      schema_ref: "CreateOrderRequest"
    request:
      method: POST
      url: "${base_url}/api/orders"
      body:
        product_id: "PROD-001"
        quantity: 9999            # 超过库存
    assertions:
      - type: status
        expected: 400
      - type: json_path
        path: "$.error.code"
        expected: "INSUFFICIENT_STOCK"

boundary_expansion:               # 边界引擎自动展开
  parameters:
    - name: quantity
      type: integer
      constraints: { min: 1, max: 9999 }
      generated_cases:
        - { value: 0, expected: 400, reason: "边界下" }
        - { value: 1, expected: 200, reason: "边界最小值" }
        - { value: 9999, expected: 400, reason: "边界超库存" }
        - { value: 10000, expected: 400, reason: "边界上" }
        - { value: -1, expected: 400, reason: "异常负数" }
        - { value: null, expected: 400, reason: "异常null" }

impact_analysis:                  # TIA: 哪些代码变更会影响此测试
  depends_on:
    - file: "src/orders/service.py"
      functions: ["create_order", "check_inventory"]
    - file: "src/inventory/manager.py"
      functions: ["reserve_stock"]
```

## 八、核心命令设计

```bash
# ===== 测试设计 =====
testgen design                       # 打开可视化设计器(Web UI)
testgen create "用户登录测试"        # 自然语言创建测试
testgen record browser               # 开始浏览器录制
testgen record api --app "python main.py"  # API流量录制

# ===== 智能分析 =====
testgen analyze --impact HEAD~1      # 分析变更影响范围
testgen analyze --flaky              # 扫描Flaky测试
testgen analyze --health             # 测试健康度报告

# ===== 执行 =====
testgen run                          # 全量运行
testgen run --smart                  # 智能选择(TIA)运行
testgen run --only-flaky             # 仅重跑已隔离的Flaky测试

# ===== 维护 =====
testgen heal                         # 自愈失败的测试
testgen quarantine <test-id>         # 手动隔离Flaky测试
testgen unquarantine <test-id>       # 恢复隔离的测试

# ===== 报告 =====
testgen report                       # 生成测试报告
testgen report --trend               # 趋势分析
```
