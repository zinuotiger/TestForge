### 测试设计器能力

| 能力 | 说明 |
|------|------|
| **可视化编排** | 拖拽式编排测试步骤：HTTP请求、数据库查询、消息队列、脚本、断言、等待 |
| **智能边界生成** | 选中参数 → 自动分析类型 → 推荐等价类/边界值/异常值组合 |
| **自然语言→测试** | 输入"测试用户登录功能的各种异常情况" → AI自动生成测试步骤和数据 |
| **录制回放** | 浏览器操作录制 → 自动转为测试步骤；API流量录制 → 自动转为API测试 |
| **参数化** | 一个测试模板 + 多组数据 = 批量生成用例 |
| **断言库** | 状态码/响应体/响应时间/数据库状态/消息队列/文件存在 等内置断言 |
| **变量系统** | 步骤间数据传递、环境变量、全局变量、随机数据生成 |

## 六、边界测试智能引擎（核心差异化）

用户不需要手动写边界值，TestForge 自动分析并生成：

```
参数分析 → 类型推断 → 等价类划分 → 边界值计算 → 异常值推荐 → 组合优化

示例: 函数 register(username: string, age: int, email: string)

┌──────────┬──────────────┬──────────────┬──────────────┬──────────────┐
│ 参数     │ 有效等价类   │ 无效等价类   │ 边界值       │ 异常值       │
├──────────┼──────────────┼──────────────┼──────────────┼──────────────┤
│ username │ 3-20位字母   │ <3位         │ 3位(最小)    │ null         │
│          │ 数字下划线   │ >20位        │ 20位(最大)   │ 空字符串     │
│          │              │ 特殊字符     │              │ SQL注入      │
│          │              │              │              │ XSS脚本      │
├──────────┼──────────────┼──────────────┼──────────────┼──────────────┤
│ age      │ 1-150整数    │ <1           │ 1(最小)      │ null         │
│          │              │ >150         │ 150(最大)    │ -1           │
│          │              │ 非整数       │ 0(边界下)    │ 999999       │
│          │              │              │              │ "abc"        │
├──────────┼──────────────┼──────────────┼──────────────┼──────────────┤
│ email    │ user@dom.com │ 无@          │ 单字符名@    │ null         │
│          │              │ 无域名       │ 超长邮箱     │ 空字符串     │
│          │              │ 无TLD        │              │ 多@符号      │
└──────────┴──────────────┴──────────────┴──────────────┴──────────────┘

自动生成: 3×4×3 = 36 个边界测试用例
优化后 (Pairwise): ~12 个用例覆盖所有边界组合
```

## 七、四种测试设计入口

```
入口1: 界面设计 (拖拽+配置)
  用户打开Web UI → 测试设计器 → 拖拽编排步骤 → 配置断言 → 保存
  适合: QA、业务人员

入口2: 自然语言 (AI理解)
  用户输入: "测试商品搜索功能，包括空搜索、特殊字符、超长关键词、SQL注入"
  → AI解析意图 → 自动生成测试步骤+边界数据 → 用户确认 → 保存
  适合: 所有人

入口3: 录制回放 (操作转测试)
  浏览器操作: 打开页面 → 输入 → 点击 → 等待 → 自动生成测试步骤
  API流量: Keploy eBPF录制网络流量 → 自动生成API测试
  适合: 已有系统快速补测试

入口4: 代码导入 (已有测试导入)
  导入已有pytest/Jest/JUnit测试 → 在界面中管理、可视化结果
  适合: 已有测试资产的团队迁移
```

## 八、技术选型调整

| 组件 | 原方案 | v2.0 调整 | 理由 |
|------|--------|-----------|------|
| **前端界面** | ❌ 无(纯CLI) | ✅ React + TypeScript | 需要可视化测试设计器、拖拽编排 |
| **后端API** | ❌ 无 | ✅ FastAPI + WebSocket | 实时推送执行进度、日志流 |
| **测试步骤DSL** | ❌ 无 | ✅ YAML/JSON + 自研执行引擎 | 统一的测试描述语言 |
| **浏览器自动化** | ⚠️ Playwright(仅代码生成) | ✅ Playwright(全功能) | 录制+回放+UI测试执行 |
| **数据库** | SQLite | SQLite(本地) + PostgreSQL(服务端) | 多用户协作需要 |
| **实时通信** | ❌ 无 | ✅ WebSocket | 执行进度实时推送 |
| **任务队列** | ❌ v2.0 | ✅ Celery + Redis(v1.0就上) | 异步执行测试套件 |
| **自然语言处理** | ❌ 无 | ✅ LLM + Prompt工程 | 自然语言→测试用例 |

### 测试用例DSL

```yaml
# 一个完整的测试用例
id: "tc_login_001"
name: "用户登录-密码错误3次锁定"
type: functional          # functional | boundary | unit | api | e2e | performance
tags: [login, security, P0]
description: "验证连续3次密码错误后账户被锁定"

variables:
  base_url: "https://api.example.com"
  test_user: "testuser@example.com"

steps:
  - id: step1
    type: http_request
    description: "第1次错误密码登录"
    request:
      method: POST
      url: "${base_url}/auth/login"
      body:
        email: "${test_user}"
        password: "wrong_password_1"
    extract:
      status: "$.status"
    assertions:
      - type: equals
        actual: "${status}"
        expected: 401
      - type: contains
        actual: "$.body.message"
        expected: "密码错误"

  - id: step2
    type: http_request
    description: "第2次错误密码登录"
    request:
      method: POST
      url: "${base_url}/auth/login"
      body:
        email: "${test_user}"
        password: "wrong_password_2"
    assertions:
      - type: equals
        actual: "$.status"
        expected: 401

  - id: step3
    type: http_request
    description: "第3次错误密码登录"
    request:
      method: POST
      url: "${base_url}/auth/login"
      body:
        email: "${test_user}"
        password: "wrong_password_3"
    assertions:
      - type: equals
        actual: "$.status"
        expected: 423
      - type: contains
        actual: "$.body.message"
        expected: "账户已锁定"

boundary_analysis:            # 边界分析配置 ⭐
  parameters:
    - name: password
      type: string
      constraints:
        min_length: 6
        max_length: 128
      test_values:
        valid: ["correct_password"]
        boundary: ["", "a", "12345", "a"*128, "a"*129]
        exception: [null, "'; DROP TABLE users;--", "<script>alert(1)</script>"]
```

## 十二、用户使用流程

```
1. 启动: docker compose up → 打开浏览器 http://localhost:3000

2. 创建项目: 输入项目名 + 目标URL + 语言/框架

3. 设计测试 (3种方式):
   A) 界面设计: 测试设计器 → 添加步骤 → 配置断言 → 保存
   B) 自然语言: 输入"测试搜索为空时返回提示" → AI生成 → 确认
   C) 录制: 打开录制器 → 操作浏览器 → 自动生成步骤

4. 边界增强: 选中参数 → 点击"边界分析" → 预览生成的边界用例 → 确认

5. 代码补全: TestForge自动扫描源码 → 建议补充单元测试 → 确认生成

6. 执行: 点击"运行" → 实时查看进度/日志 → 全部跑通 ✅

7. 分析: 查看通过/失败详情 → AI分析失败原因 → 一键修复建议

8. 集成: 导出JUnit XML → 配置GitHub Actions → CI中自动运行
```
